# 需求理解与假设说明

> 本文档为强制交付物，随源码一起提交。题目中部分规则故意未给出精确数值，这是有意为之——现实中产品经理的需求经常不完整，工程师需要主动判断、主动假设。

## 一、关键留白点补全（九项必须覆盖）

### ① 分级审批金额阈值

| 层级 | 金额阈值 | 设定依据 |
|---|---|---|
| 一级审批（主管级） | 金额 < 100 元 | 常规异常处理金额较小，由一线主管快速决策即可 |
| 二级审批（经理级） | 金额 ≥ 100 元 | 超过该金额的赔付/追偿需要更高层级审核，控制财务风险 |
| 品控异常（扫描触发） | 自动进入二级审批 | 品控异常属于上游质量事故，可能涉及批量问题，直接拉高审批层级 |

**设计要求**：阈值必须实现为**后台可配置项**，不能硬编码为 `if (amount > 5000)`。设计为规则引擎驱动（参考 V2 设计理念），支持按金额区间配置审批层级。

### ② 审批超时时长

| 审批环节 | 超时时长 | 设定依据 |
|---|---|---|
| 一级审批待处理 | 48 小时（2 个工作日） | 给审批人留出合理处理时间，同时避免工单长时间积压 |
| 二级审批待处理 | 72 小时（3 个工作日） | 经理级审批涉及更复杂的判断，预留稍长处理时间 |
| 超时处理方式 | 自动升级至上一级 | 待审批超时直接升级到二级审批；二级审批超时自动驳回（`closed_timeout`） |

**设计要求**：各环节超时时长独立配置，不能共用一个配置值。

### ③ 重提次数上限

| 规则 | 数值 | 设定依据 |
|---|---|---|
| 最大重提次数 | 3 次 | 允许合理纠错，但防止无限反复占用审批资源 |
| 超过上限后处理 | 自动关闭工单，状态记为 `closed_rejected_final` | 避免工单永久悬空，需强制终结 |
| 计数重置 | 不重置，每个工单有独立计数器 | 简洁清晰 |

### ④ 物流异常类型映射（各类型 → 下游动作）

| 异常类型 | 是否涉及赔付 | 库存联动 | 是否需要重新发货 | 说明 |
|---|---|---|---|---|
| 丢件 | 是（赔付客户） | 扣减库存 | 是（如需补发） | 确认丢件 → 理赔 + 可选补发 |
| 破损 | 是（赔付客户） | 退货入库（破损件）+ 扣减（补发件） | 是 | 根据破损程度决定全额/部分赔付 |
| 客户拒收 | 看情况 | 退货入库 | 否（默认） | 拒收原因决定是否赔付（如质量问题则赔付） |
| 超时未签收 | 否（默认） | 无联动 | 否 | 通常与客户协商解决，可转拒收或丢件 |
| 收货地址错误 | 否 | 无联动 | 是 | 更正地址后重新发货，不涉及赔付 |

### ⑤ 角色权限划分

| 角色 | 权限范围 | 说明 |
|---|---|---|
| **上报人**（operator） | 上报异常工单、查看自己上报的工单 | 不能审批自己上报的工单（自批自核防御） |
| **一级审批人**（approver_l1） | 审批金额 < 阈值的工单；仅能看到自己权限范围内的工单 | 不能跨商户/跨仓库审批他人运单 |
| **二级审批人**（approver_l2） | 审批金额 ≥ 阈值的工单；处理升级工单 | 层级更高，权限范围更广 |
| **品控主管**（qc_supervisor） | 品控误判快速放行、查看品控类工单 | **误判快速放行仅品控主管可操作**，后端接口必须校验角色 |
| **管理员**（admin） | 系统配置（阈值、超时时长、品控规则等）、角色分配 | 不直接参与审批流程 |

**权限重叠规则**：
- 一个人可以同时担任多个角色（如既是品控主管又是一级审批人），但同一工单上的操作权限不叠加——审批链上的角色不能既是审批人又是上报人
- 品控主管的"快速放行"权限与审批人权限不重叠——快速放行绕过审批流程，是独立于审批链的特殊操作
- 审批人列表只展示对应层级范围内的工单，后端接口需做权限校验（非仅前端隐藏）

### ⑥ V2 数据同步频率与一致性策略

| 策略项 | 选择 | 理由 |
|---|---|---|
| **同步触发方式** | 混合模式 | — |
| └ 异常上报时 | 实时拉取最新运单信息（强制校验运单存在性） | 上报是关键动作，必须用最新数据 |
| └ 定时批量同步 | 每 30 分钟全量/增量同步一次 | 保持快照表基本新鲜，降低接口压力 |
| └ 工单详情页 | 优先实时拉取；实时接口失败时回退到本地缓存 | 兼顾新鲜度和可用性 |
| **V2 数据变更感知** | 实时校验 + 定期对账 | — |
| └ 上报时 | 实时调用接口，以 V2 返回为准 | 保证上报那一刻数据的准确性 |
| └ 对账策略 | 工单审批通过执行联动动作前，再次通过接口拉取最新运单数据（尤其是金额） | 防止"审批期间金额被更正"导致的赔付错误 |
| └ 差异处理 | 若最新数据与工单创建时数据不一致，记录差异日志并暂停自动执行，转为人工确认 | 金额变更可能意味着需要重新定级审批 |

### ⑦ 品控暂扣超时时长

| 配置项 | 数值 | 设定依据 |
|---|---|---|
| 品控暂扣超时时长 | 4 小时 | 品控暂扣 = 货物压仓产生运营成本，超时时长必须远短于审批超时（48/72 小时）。4 小时是仓储物流中"异常件快速处理"的行业参考值：一个班次（4-8 小时）内必须决定货物去向，超时则自动升级 |
| 超时处理方式 | 强制升级至二级审批 | 品控异常自动进入二级审批环节，跳过一级审批加速处理 |
| 独立性说明 | 与审批超时独立配置，不共用值 | 品控暂扣与工单审批是两套独立状态机，超时时长反映不同维度的业务压力：暂扣 = 仓储运营成本驱动，审批 = 管理效率驱动 |

### ⑧ 品控规则触发阈值

| 规则类别 | 子类型 | 触发条件 | 严重度 | 自动创建工单 | 审批层级 |
|---|---|---|---|---|---|
| 数量不符 | 短少 | 实际数量 < 应到数量，差额 ≥ 5% | 中 | 是 | 一级审批 |
| 数量不符 | 短少严重 | 差额 ≥ 20% | 高 | 是 | 二级审批 |
| 数量不符 | 超出 | 实际数量 > 应到数量，差额 ≥ 5% | 低 | 是 | 一级审批 |
| 外观破损 | 轻微破损 | 包装破损但货物完好，破损 ≤ 1 级 | 低 | 否（仅记录） | — |
| 外观破损 | 中度破损 | 货物表面损伤但功能正常，破损 ≥ 2 级 | 中 | 是 | 一级审批 |
| 外观破损 | 严重破损 | 货物结构性损坏不可用，破损 ≥ 3 级 | 高 | 是 | 二级审批 |
| 规格不符 | — | SKU 尺寸/重量/型号与系统记录偏差 ≥ 10% | 高 | 是 | 二级审批 |
| 标签错误 | — | 标签信息（品名/批次/有效期）与系统不一致 | 高 | 是 | 二级审批 |
| 批次异常 | — | 批次号不在允许范围内或已过期 | 中 | 是 | 一级审批 |

**设计要求**：
- 所有触发条件（阈值百分比、破损等级、偏差范围）实现为**后台可配置项**，不可硬编码
- 规则引擎执行过程必须可追溯：记录命中规则 ID、触发条件实际值、判定依据
- 规则是否自动创建工单、自动进入哪级审批均为可配置属性

### ⑨ 品控主管角色权限边界

| 权限项 | 是否允许 | 说明 |
|---|---|---|
| 品控误判快速放行 | ✅ 允许 | 核心权限，绕过完整审批流程，直接解锁批次、关闭工单 |
| 操作一级/二级审批 | ❌ 不允许（默认） | 快速放行是独立于审批链的操作，与审批权限不重叠。如需同时担任审批人，需单独分配审批角色 |
| 查看品控类工单 | ✅ 允许 | 查看所有品控相关的工单和扫描记录 |
| 查看物流类工单 | ❌ 不允许（默认） | 品控主管的权限范围限定在品控类工单 |
| 修改品控规则配置 | ✅ 允许 | 调整触发阈值、严重度等级等规则参数 |
| 兼任审批人 | ✅ 允许 | 但同一工单上不得同时执行快速放行和审批操作 |
| 兼任上报人 | ✅ 允许 | 但不能对自己上报的工单执行快速放行（自放自核防御） |

**快速放行操作要求**：
- 必须填写复核原因，留痕记录（谁、何时、为什么放行）
- 不允许静默放行（无原因 / 原因为空）
- 后端接口必须校验角色权限

## 二、额外假设说明

### 2.1 库存数据策略

V3 的库存表为 V3 自有，不直接连接 V2/WMS 数据库。库存变更通过以下方式实现：
- 审批通过后的库存联动（退货入库 / 重新发货扣减），在 V3 本地库存表操作
- 若需与外部 WMS 同步，通过接口调用实现，遵循同样的鉴权和超时重试策略
- 库存变更记录保留审批记录 ID 作为外键，可追溯

### 2.2 多租户假设

若 V2 数据本身不区分租户，则 V3 采用**单租户假设**。如需支持多租户，需在以下表中添加 `tenant_id` / `owner_id` 字段：
- 运单本地快照表
- 异常工单表
- 审批记录表

并在所有查询中增加租户隔离的 `WHERE` 条件，上报人只能对自己所属仓库/商户的运单发起异常上报。

### 2.3 同运单不同类型异常并存

同一运单存在未关闭的品控工单期间，若又发生物流异常（如丢件），**允许并存**。理由：品控异常（发货前）和物流异常（发货后）是不同阶段的问题，各自独立处理。但同一运单同类型未关闭工单**不允许重复上报**。

### 2.4 审批状态与下游联动一致性

审批通过后执行联动动作（赔付/库存/退仓/重采购），采用**数据库事务**保证一致性：
- 审批状态变更为"执行中"、赔付记录生成、库存变更在同一事务内完成
- 若部分操作失败，整体回滚，记录失败日志并触发人工介入
- 赔付记录、库存变更记录必须保留审批记录 ID 外键，保证可追溯

### 2.5 品控暂扣与批次锁定

- 品控暂扣期间，该批次 SKU 的库存记录标记为锁定状态，不可被其他运单引用
- 工单关闭（审批通过执行完成 / 拒绝超过重提上限 / 快速放行）时，批次状态同步解锁
- 批次解锁与工单关闭操作在同一事务内完成，不出现"工单已完成但批次仍锁定"的中间态
- 若选择"退回供应商"或"重新采购"，批次出库/作废操作也在同一事务内完成

### 2.6 并发冲突处理

两个审批人同时打开同一条待审批工单时，采用**乐观锁**机制：
- 审批提交时，比较工单当前状态和审批人打开时的状态
- 若状态已变更（已被他人处理），给出明确提示："该工单已被处理，请刷新"，拒绝操作
- 基于工单当前状态做前置校验，配合数据库行级锁或版本号字段

### 2.7 审批人离职兜底

- 审批人账号被禁用后，后台任务扫描该审批人名下所有待审批工单
- 自动转交给管理员指定的备用审批人，或上浮至上一级审批人
- 操作记录写入审批记录表，注明"因审批人已禁用自动转交"

### 2.8 超时自动流转机制

- 需实现后台定时任务（如 `cron` / `scheduler`），每 5 分钟扫描一次超时工单
- 超时工单按配置规则自动执行升级或驳回操作
- 操作记录写入审批记录表，操作人标记为"系统自动"

## 三、向产品经理的提问清单（体现主动澄清意识）

1. 品控异常中"降级处理"的降级标准是什么？哪些客户/渠道可以接受降级商品？
2. 品控主管快速放行后，是否需要对放行率等指标做统计？是否需要二次审核？
3. 同一运单有多个 SKU，部分通过、部分异常时，是通过的 SKU 先出库还是整单暂扣？
4. 赔付金额的计算标准是什么？运单申报价值还是实际采购成本？物流公司和供应商的赔偿标准是否不同？
5. 审批人是否可以委派/转交审批权限？如果可以，转交后超时时长是否重置？
6. 超时自动驳回的工单，上报人是否可以直接重新提交（不经过修改）？
7. 品控暂扣超时后强制升级二级审批，如果二级审批人也超时，是否继续向更高层级升级？
8. V2 运单数据中的金额是含税还是不含税？审批阈值按含税还是不含税计算？
