## 1. 如果题目直接给出了精确的审批金额阈值和超时时长，你的架构设计会有什么不同？

如果题目直接给出精确数值，架构设计上会有以下变化：

**表结构层面**：可能不会单独设计"审批规则配置表"和"品控规则表"，而是将阈值直接写到工单表的字段里，或者作为常量放在配置文件中。但正因题目故意留白，才促使我在设计时将阈值全部抽象为可配置的规则引擎，带来了更好的可扩展性。

**代码层面**：如果数值固定，审批层级判断可能就是简单的 `if (amount > 5000)` 硬编码。而可配置的设计需要查表、匹配规则区间，代码复杂度更高，但换来的是业务人员可自助调整、无需改代码上线的好处。

**本质上**：题目不给数值反而让我做了更"正确"的架构——规则引擎 + 配置表。如果一开始就给了精确值，我大概率会偷懒硬编码，后续产品经理改阈值时就要改代码、走发布流程。所以这个考点确实有意义。

## 2. 你认为本题里"审批状态变更"与"库存/赔付联动"之间的一致性，最大的风险点在哪里？如果不用数据库事务，你会怎么处理？

**最大风险点**：审批状态变为"执行中"之后、库存/赔付操作成功之前的时间窗口。如果在这个窗口内系统崩溃或网络中断，就会出现"审批显示已通过且进入执行中，但库存没有扣减、赔付记录没有生成"的中间态。而且这个中间态很难被自动发现——从界面上看一切都正常，直到财务对账时才发现赔付款没出去。

**如果不用数据库事务**，我会采用以下补偿方案：

1. **Outbox 模式**：审批通过时，不直接操作库存/赔付表，而是先将待执行的动作写入一张"执行任务队列表"（`execution_task`），状态为 `pending`。库存和赔付的变更由后台 worker 消费队列逐条执行。
2. **幂等执行**：每个执行任务有唯一 `task_id`，worker 执行前检查是否已处理，防止重复执行。
3. **失败重试 + 告警**：worker 执行失败时指数退避重试（最多 3 次），最终仍失败则标记为 `failed` 并触发告警，由人工介入排查。
4. **补偿事务（Saga）**：如果执行链条中某一步失败（比如库存扣减成功但赔付记录写入失败），需要执行补偿操作将已成功的步骤回滚（库存加回）。

这种方式比数据库事务更复杂，但它在分布式场景下（库存和赔付不在同一数据库/服务）更适用。单库单服务的场景下，数据库事务仍然是最简单也最可靠的选择。

## 3. 面对一句话级别的模糊需求，你倾向于"先问清楚再动手"还是"先做出一个版本再迭代"？为什么？

我的倾向是**两者结合**，但以"先问"为起点：

1. **先用最短时间列出疑问清单**：把模糊点一条条写下来（就像本题的"向产品经理提问清单"），哪怕产品经理没回复，至少自己心里有数哪些地方是"自己做主"的、哪些是"需要确认的"。
2. **对阻塞性问题优先沟通**：比如"赔付金额计算标准"这种直接影响数据库字段设计的，如果不问清楚就做，表结构错了后面的改动成本极高。
3. **对非阻塞性模糊点先做合理假设、再迭代**：比如超时时长、审批阈值这类"可以后续配置"的，先设一个合理默认值让系统跑起来，后续根据实际业务数据调整。

**为什么不选"直接做一个版本再迭代"**：太危险。数据库设计、状态机设计这类底层决策一旦定错方向，迭代的成本不是改几行代码，而是数据迁移 + 历史数据兼容 + 接口变更的三重代价。

**为什么不选"问清楚每一个细节再动手"**：现实中产品经理可能也没想清楚所有细节，或者没时间回复。工程师需要有权衡能力——哪些必须确认、哪些可以假设。

**本题的实践**：我在《需求理解与假设说明》中为九个关键留白点给出了明确假设并附依据，同时列出了我会向产品经理确认的八个问题。这就是"先假设 + 留痕 + 待确认"的方式。

## 4. 如果产品经理后续要求新增第六种异常类型（比如"客户投诉服务态度"，不涉及货品本身、不涉及库存联动），你现在的表结构和状态机需要改动哪些地方？改动成本大不大？

**需要改的地方**：

| 改动项 | 改动内容 | 成本 |
|---|---|---|
| 异常类型枚举 | `exception_type` 字段新增 `service_complaint` | **低**：加一个枚举值即可 |
| 异常类型→下游动作映射 | 《假设说明》文档中新增一行映射：客户投诉 → 不赔付、不联动库存、不需重新发货 | **低**：纯配置/逻辑判断 |
| 执行联动模块 | 新增执行动作类型：`none`（无联动），审批通过后直接进入"已完成" | **低**：需要一个空分支处理 |
| 状态机 | 工单状态机本身**不需要改动**，`投诉` 类型的工单走同样的待审批→一审→通过→已完成流程，只是"执行中"环节为空 | **零**：现有状态机直接兼容 |
| 赔付记录表 | **不需要改动**，投诉类工单审批通过时不生成赔付记录 | **零** |
| 库存表 | **不需要改动**，不涉及货品 | **零** |

**结论**：改动成本很小。原因是"服务态度投诉"这种异常是现有系统的**子集**——它不需要任何新能力，只是不需要部分现有能力（赔付、库存联动）。系统架构支持"部分执行动作可为空"的设计（异常类型映射表中的"是否赔付/是否联动库存/是否重新发货"字段天然支持全否的情况），所以基本只需要加枚举值 + 配置即可。

**如果反过来**，新增的异常类型要求的是系统完全不具备的能力（比如"需自动对接保险公司理赔系统"），那改动成本就会大得多。所以判断改动成本的关键维度不是"异常类型多了一种"，而是"新类型需要的执行动作在不在现有系统的能力范围内"。

## 5. 如果工单量从 200 条增长到 20 万条，你的系统里最先撑不住的环节是哪个（列表查询？定时任务全表扫描？接口同步积压？）？你会怎么处理？

**最先撑不住的环节：三个都可能出问题，但崩溃顺序大概率是**：

### 第一：定时任务全表扫描（最危险）

超时检测定时任务每 5 分钟扫描全部未关闭工单。20 万条工单中即使只有 10% 处于未关闭状态（2 万条），每次扫描就是一次全表 `WHERE status IN (...)` 查询。加上品控暂扣超时扫描，两个定时任务叠加，数据库压力骤增。

**处理方案**：
- 在 `status` + `created_at` 上建联合索引，避免全表扫描
- 改为"只扫描即将超时的工单窗口"：比如只查 `updated_at < NOW() - (timeout - 5min) AND status IN (...)`，而不是扫描全部
- 考虑用延迟队列替代定时扫描：工单进入待审批状态时，同时投递一条延迟消息（如 Redis `ZSET` + `TTL`），到期自动触发超时处理，彻底消除定时扫描

### 第二：列表查询（分页不彻底时）

用户打开工单列表默认 `SELECT * FROM tickets ORDER BY created_at DESC`，没有复合索引的情况下 20 万行全表排序。

**处理方案**：
- 在常用筛选列上建索引（`status`、`exception_type`、`assignee_id`、`created_at`）
- 强制分页，禁用"不分页全量拉取"
- 统计数据（各状态工单数）改用缓存或物化视图，不从明细表实时 `COUNT`

### 第三：接口同步积压

30 分钟一次的全量/增量同步，数据量大时可能超过同步周期，导致永远追不上。

**处理方案**：
- 增量同步改为游标模式，分批拉取
- 同步频率可以根据数据量自动调整（数据多时降低频率）
- 快照表不要全量覆盖，改为 upsert，减少写入压力

### 小结

综合来看，定时任务全表扫描是最容易被忽略但杀伤力最大的——因为它是后台静默运行的，不出问题不会被注意到，一旦数据库被拖垮就是全局性的。

## 6. 你在给 V2 新增对外接口时，如何保证不破坏 V2 现有的调用方？如果 V2 的运单金额字段类型从 `int` 改成 `decimal`，你会如何处理 V3 侧的兼容？

### 保证不破坏现有调用方

1. **URL 路径版本化**：新增接口放在 `/api/v2/` 下，与旧接口 `/api/v1/` 隔离。旧接口原封不动，新接口独立路由，互不干扰。
2. **字段"只增不减不改"原则**：新增字段标记为 `optional`；已有字段不删（即使新业务不需要）；已有字段类型和语义不变。
3. **接口契约测试**：在 V2 侧对新接口做契约测试（consumer-driven contract test），验证接口输出结构与约定一致。
4. **灰度发布**：先在预发布环境用 V3 的 staging 实例调用验证，确认无异常后切生产流量。V2/V3 两边的切换用一个 feature flag 控制，出问题可以瞬间回退到不调用新接口的状态。
5. **监控 + 告警**：新接口上线后关注错误率、响应时间 P99，设置阈值告警。

### V2 金额字段从 `int` 改成 `decimal` 的处理

这是一个典型的 **breaking change**，需要在 V3 侧做好兼容：

1. **格式转换层**：V3 不直接在业务代码中使用接口返回的原始数值，而是在接入层（API client / adapter）做一层类型适配——将 `int`、`decimal`、`string` 统一转为内部使用的标准格式（如 `BigDecimal` 字符串或高精度数值类型）。
2. **宽松解析**：V3 的 JSON 反序列化不使用严格的类型绑定。金额字段用 `number | string` 接收，运行时判断类型后转换。
3. **响应结构监控**：在接口同步日志表中记录每次调用的响应结构哈希（JSON Schema hash），当字段类型发生变化时触发告警，运维能第一时间感知到 V2 侧有 breaking change。
4. **对接沟通流程**：V2 做字段变更前应通知 V3 侧，双方约定过渡期。过渡期内 V2 同时返回旧格式（`int`）和新格式（`decimal`），V3 优先读新格式、回退读旧格式。过渡期结束后 V2 下线旧字段。
5. **测试覆盖**：在 V3 的接口调用测试中，增加对两种数据格式的兼容性用例，避免上线时才发现解析报错。
