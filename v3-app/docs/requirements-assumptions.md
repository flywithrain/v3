# 需求理解与假设说明

本文档覆盖考试题目中未给出精确数值的九项留白规则，说明 V3 的设定值及其依据。

## ① 分级审批金额阈值

| 规则 | 默认值 | 依据 |
|---|---|---|
| 物流异常损失 ≤ 1000 元 | 一级审批 | 低金额理赔应快速处理，减少客户等待 |
| 物流异常损失 > 1000 元 | 二级审批 | 高金额需加强控制，参考电商行业常见理赔审批线 |
| 任意异常 severity = high | 强制二级 | 高严重度不论金额均需高层把关 |
| 品控异常 | 默认二级 | 涉及批次锁定和供应商追偿，风险集中 |

**可配置性**：阈值存储在 `approval_rules` 表的 `condition_config` jsonb 中（`amountLte` / `amountGt`），admin 可在 `/rules/approval` 页面调整，不硬编码。

## ② 审批超时时长

| 节点 | 超时 | 超时动作 | 依据 |
|---|---|---|---|
| 待审批未分派 | 4h | 自动进入一级队列 | 避免工单卡在初始态 |
| 一级审批中 | 8h | 自动升级二级 | 一级应在工作日内完成 |
| 二级审批中 | 24h | 自动驳回 `auto_rejected_timeout` | 二级需更慎重但不无限等待 |

**理由**：8 小时覆盖一个工作日；24 小时覆盖一个自然日。超时后管理员可重新打开。

## ③ 重提次数上限

- 拒绝后最多允许重新提交 **2 次**
- 第 3 次被拒绝后自动关闭，状态为 `closed_rejected_limit`
- 管理员可基于审计原因重新打开

**理由**：2 次重提给予上报人补充信息的机会，但不允许无限循环。参考 Jira 等工单系统的常见重提上限。

## ④ 物流异常类型映射

| 异常类型 | 默认动作 | 赔付客户 | 库存联动 |
|---|---|---|---|
| 丢件 lost | 客户理赔 + 重新发货 | 是 | 重新发货扣减库存 |
| 破损 damaged | 客户理赔 + 退货入库/重新发货 | 是 | 退货入库增加可疑库存，重新发货扣减 |
| 客户拒收 rejected | 退货入库 | 否（除非人工补偿） | 退货入库 |
| 超时未签收 timeout_unsigned | 补偿券/部分理赔 | 可选 | 默认不联动 |
| 地址错误 address_error | 修改地址后重新发货 | 否 | 重新发货扣减库存 |

**理由**：丢件/破损涉及赔付并需回滚原运单库存；地址错误只需重新发货不涉及赔付；拒收通常退货入库即可。

## ⑤ 角色权限划分

| 角色 | 能做 | 不能做 |
|---|---|---|
| operator 操作员 | 手工上报物流异常 | 审批（含自己的）、扫描 |
| warehouse_operator 仓管员 | 扫描录入、查看扫描记录 | 快速放行、审批 |
| qc_supervisor 品控主管 | 快速放行、一级审批 | 审批自己发起/放行过的工单 |
| level1_approver 一级审批人 | 处理一级审批 | 二级审批、上报 |
| level2_approver 二级审批人 | 处理二级审批和升级工单 | 上报 |
| admin 管理员 | 规则配置、兜底转交、监控 | — |
| auditor 审计员 | 只读查看全部 | 任何写操作 |

**自批自核**：上报人不能审批自己提交的工单，后端接口强制校验。同一用户可拥有多角色，但自批自核规则始终生效。

## ⑥ V2 数据同步频率与一致性策略

- **实时校验**：发起异常上报、扫描 SKU 时实时调用 V2 API
- **定时同步**：每 15 分钟增量同步最近 7 天有变更的运单快照
- **本地缓存**：列表/详情优先展示本地快照，详情页提供「刷新 V2」按钮
- **降级**：V2 不可用时只读展示缓存并标注「数据可能非最新」
- **一致性处理**：V2 数据变更时通过实时校验感知差异，工单详情页标注提示

**理由**：关键操作（上报/扫描）必须实时校验，非关键展示用缓存减少 V2 压力。15 分钟是时效性与 API 负载的平衡点。

## ⑦ 品控暂扣超时时长（独立于审批超时）

- 品控暂扣超时：**2 小时**
- 超时动作：强制升级二级审批

**为何独立于审批超时**：品控暂扣意味着货物压仓，直接造成库存占用和履约延迟。货物压仓的运营成本远高于审批等待成本，因此暂扣超时（2h）应远短于审批超时（8h/24h）。

**数值依据**：2 小时覆盖一个紧急处理窗口，品控主管应在此时限内决定放行或升级。

## ⑧ 品控规则触发阈值

| 品控子类型 | 判定条件 | 默认严重度 | 依据 |
|---|---|---|---|
| 数量不符 | 实扫数量与 V2 不一致，或差异率 ≥ 5% | medium | 5% 容差覆盖正常盘点误差 |
| 外观破损 | 破损等级 ≥ 2 触发；≥ 4 直接 high | medium/high | 2 级为可见破损需处理，4 级为严重破损 |
| 规格不符 | 实扫规格与 V2 SKU 规格不一致 | high | 规格不符影响履约正确性 |
| 标签错误 | 标签 SKU 与运单 SKU 不一致 | high | 标签错误可能发错货 |
| 批次异常 | 批次号命中召回/禁售/过期规则 | high | 涉及合规风险 |

**可配置性**：阈值存储在 `qc_rules` 表的 `condition_config` jsonb 中（如 `diffThresholdPct: 5`、`damageLevelMin: 2`），admin 可在 `/rules/qc` 页面调整。规则执行过程记录命中 `qc_rule_id` 和判定依据。

## ⑨ 品控主管角色权限边界

| 权限项 | 说明 |
|---|---|
| 误判快速放行 | **仅**品控主管可操作，绕过完整审批流程 |
| 留痕要求 | 必须填写复核原因，写入 `audit_logs`，不允许静默放行 |
| 审批权限 | 可作为一级审批人，但不能审批自己发起或自己快速放行过的工单 |
| 权限重叠 | 与 level1_approver 有重叠（均可一级审批），但快速放行是独占权限 |
| 多角色兼任 | 同一用户可同时拥有 qc_supervisor + level2_approver，但自批自核规则始终生效 |

**理由**：品控主管是批次质量的最终责任人，应有权快速纠偏（误判放行），但必须有留痕以防滥用。与其他审批角色部分重叠便于灵活分派，但关键权限（快速放行）独占以确保问责清晰。

## 向产品经理提问清单

以下是在开发过程中识别出的、如果能向产品经理提问会进一步澄清的问题：

1. **赔付金额计算**：物流异常的赔付金额是由上报人手工填写，还是应基于商品价值自动计算？如果自动计算，商品价值从哪个字段获取？
2. **多 SKU 联动**：一个运单有多个 SKU，审批通过后的执行联动应该对所有 SKU 操作还是仅对异常 SKU？当前仅对首个 SKU 操作。
3. **供应商追偿对账**：品控追偿记录的对账方式是只做记录还是需要对接财务系统？追偿金额如何确定？
4. **超时通知**：工单即将超时是否需要接入钉钉/企微机器人通知？当前仅在列表中用颜色标注。
5. **工单转交**：审批人离职后待审批工单的转交，是自动转交（按角色分配）还是需管理员手动转交？当前需管理员手动操作。
6. **V2 运单金额**：V2 的 `shipments` 表无金额字段，V3 的 `estimated_amount` 由上报人填写。是否应在 V2 侧增加金额字段以支持自动计算？

## V2 二开意识

### 如果 V2 原本没有现成接口

V2 当前只有页面和 Server Actions，没有对外 API。V3 在 V2 中新增 `/api/v1/*` 只读接口时：

1. **只新增文件**：`src/app/api/v1/` 目录下新增 route 文件，不修改任何现有页面/Server Action/数据库结构
2. **接口版本策略**：统一 `/api/v1/` 前缀，与 V2 现有 `/api/*` 路由物理隔离
3. **字段向后兼容**：新增字段不删除已有字段；V3 侧对可选字段做 null 容错
4. **CORS 配置**：仅在 `next.config.ts` 追加 CORS headers，不修改现有配置
5. **鉴权独立**：新增 `src/lib/v2-api-auth.ts`，与 V2 现有鉴权逻辑完全独立

### V2 字段类型变化处理

以金额字段从 `int` 改为 `decimal` 为例：

1. V3 快照表使用 `numeric` 类型，天然兼容 int/decimal
2. 接口层使用 `String()` 传递，避免 JS number 精度问题
3. V3 通过 `raw_payload` jsonb 保留原始响应，兼容字段映射变更
4. 若字段名变更，V3 可通过 `source_version` 感知并适配
