# V3 运单全流程管理系统 —— 系统间接口文档

> 本文档对应考试题目第七项「提交要求」，说明 V3 调用 V2 的接口列表、入参出参、鉴权方式、超时与重试策略、V2 不可用时的降级方案。

---

## 1. 概述

V3 与 V2 是两个独立部署、独立数据库的系统，运单数据交互必须通过 **HTTP API** 完成，不允许 V3 直接连接 V2 数据库。V3 不直接持有完整运单主数据，而是通过调用 V2 暴露的接口获取。

## 2. 接口列表

### 2.1 校验运单是否存在 + 获取运单详情

| 项目 | 说明 |
|---|---|
| **用途** | 发起异常上报时的真实性校验；工单详情页展示运单信息 |
| **请求方式** | `GET /api/v2/waybills/{waybillNo}` |
| **入参** | `waybillNo`（运单号，路径参数） |
| **出参** | 运单详情 JSON：运单号、收件人信息、发件人信息、金额、SKU 明细列表、状态、创建时间、更新时间 |
| **鉴权** | Header `Authorization: Bearer {API_KEY}` |
| **超时** | 5 秒 |
| **重试** | 失败重试 2 次（间隔 1s / 2s），幂等性由 `GET` 语义天然保证 |

### 2.2 校验 SKU 是否归属于指定运单

| 项目 | 说明 |
|---|---|
| **用途** | 扫描录入时验证该 SKU 确实在该运单的 SKU 明细中，避免扫描到无关货物 |
| **请求方式** | `GET /api/v2/waybills/{waybillNo}/skus/{skuCode}` |
| **入参** | `waybillNo`（运单号），`skuCode`（SKU 编号） |
| **出参** | SKU 归属校验结果：`{ exist: boolean, skuInfo: { name, quantity, ... } }` |
| **鉴权** | Header `Authorization: Bearer {API_KEY}` |
| **超时** | 3 秒 |
| **重试** | 失败重试 1 次（间隔 1s） |

### 2.3 按条件查询/同步运单列表

| 项目 | 说明 |
|---|---|
| **用途** | 运单本地快照表的初始化或增量同步 |
| **请求方式** | `GET /api/v2/waybills?updatedAfter={timestamp}&page={page}&pageSize={pageSize}` |
| **入参** | `updatedAfter`（增量同步起点，ISO 8601 时间戳）、`page`、`pageSize` |
| **出参** | 分页运单列表 + 总数 |
| **鉴权** | Header `Authorization: Bearer {API_KEY}` |
| **超时** | 10 秒 |
| **重试** | 失败重试 1 次（间隔 2s） |

### 2.4 （可选加分项）异常处理结果回写 V2

| 项目 | 说明 |
|---|---|
| **用途** | 运单进入异常处理中时，回写 V2 侧标记"该运单存在未关闭异常"，避免 V2 继续按正常运单处理（如重复发货） |
| **请求方式** | `POST /api/v2/waybills/{waybillNo}/exception-flag` |
| **入参** | `waybillNo`，`exceptionStatus`（标记 / 清除），`v3TicketId` |
| **出参** | 操作结果 |
| **鉴权** | Header `Authorization: Bearer {API_KEY}` |
| **超时** | 5 秒 |
| **重试** | 失败重试 1 次 |

## 3. 鉴权方式

V3 调用 V2 接口统一使用 `Authorization: Bearer {API_KEY}` 方式鉴权。API Key 配置在 V3 的环境变量中，由 V2 侧签发和管理，不硬编码在代码中。如实现回写功能，V2 调用 V3 接口同样需要 API Key 鉴权，不允许裸奔的开放接口。

## 4. 超时与重试策略

| 策略项 | 配置 |
|---|---|
| 默认连接超时 | 3 秒 |
| 默认读取超时 | 5 秒 |
| 查询/同步类接口超时 | 10 秒 |
| SKU 校验超时 | 3 秒 |
| 最大重试次数 | 1～2 次（依接口而定） |
| 重试间隔 | 递增间隔（1s → 2s） |
| 幂等性保证 | `GET` 请求天然幂等；`POST` 回写类请求基于 `ticketId` 做业务幂等校验 |

## 5. V2 不可用时的降级方案

当 V2 服务整体不可用时，V3 不直接白屏/报错，而是采取以下降级措施：

- **工单详情页**：使用本地快照表的"上一次同步数据"继续展示运单信息，并在数据来源处明确标注 **"数据可能非最新，获取自 YYYY-MM-DD HH:mm:ss"**
- **异常上报**：V2 不可用时，**不允许发起新的异常上报**（真实性校验无法完成），前端给出明确提示："运单验证服务暂时不可用，请稍后重试"，并展示 V2 服务状态
- **恢复后**：V2 恢复后系统自动恢复正常工作，不需要人工介入修复。积压的同步任务按正常逻辑执行

## 6. 接口同步日志与可观测性

### 6.1 接口同步日志表

每次跨系统调用需生成可追踪的 **Request ID** 并写入接口同步日志表（`api_sync_log`），字段至少包含：

| 字段 | 说明 |
|---|---|
| `request_id` | 每次调用的唯一追踪 ID（UUID） |
| `call_time` | 调用时间 |
| `api_name` | 接口名称 |
| `request_params` | 入参摘要 |
| `response_status` | 响应 HTTP 状态码 |
| `duration_ms` | 耗时（毫秒） |
| `error_message` | 错误信息（如有），需区分"V2 返回 404 运单不存在"和"网络超时"等不同错误类型 |
| `success` | 调用是否成功 |

### 6.2 监控页面

提供一个简单的接口状态/同步监控页面，展示：
- 最近一次与 V2 接口同步的时间
- 同步成功率（最近 24 小时 / 总计）
- 最近的接口调用日志列表（支持按时间、接口名、成功/失败筛选）
- 通过 Request ID 可还原一次完整调用链

### 6.3 数据来源标注

工单详情页展示运单信息时，必须明确标注数据来源：
- 实时获取：**"实时获取自 V2"**
- 使用缓存：**"使用本地缓存，同步于 YYYY-MM-DD HH:mm:ss"**

## 7. V2 接口新增策略（老系统二开意识）

假设 V2 原本没有现成的对外接口，新增接口时需遵循以下原则：

- **接口版本策略**：新接口通过 URL 路径版本号区分（如 `/api/v1/...` → `/api/v2/...`），不破坏已有调用方
- **字段向后兼容**：新增字段采用"只增不减"原则，字段标记为可选，不删除已有字段，不修改已有字段类型
- **灰度上线**：新接口先在特定 V3 实例验证，确认无问题后再全量开放
- **V2 字段升级应对**：V2 接口字段类型变更时（如金额从 `int` 改为 `decimal`），V3 通过以下方式感知和应对：
  - 接口同步日志表记录每次调用的响应结构哈希，响应结构变化时可触发告警
  - V3 解析响应时使用宽松的反序列化策略（允许未知字段、容忍类型兼容转换）
  - 金额等关键字段采用字符串/高精度类型接收，由 V3 侧统一格式化
