## 2. 介入环节

### 2.1 异常上报：AI 辅助分类

- **输入**：异常描述文本 + 异常类型候选列表
- **输出**：推荐的异常类型 + 严重度初判 + 置信度
- **呈现方式**：前端展示"AI 建议：类型 = XX，置信度 = XX%"，用户可选择接受建议或手动修改
- **标注要求**：必须明确标注 **"AI 建议，需人工确认"**

### 2.2 审批环节：AI 建议审批意见（加分项）

- **输入**：当前工单信息 + 历史审批记录（最近 10 条同类异常审批结果）
- **输出**：建议审批意见（通过 / 拒绝 / 升级）+ 依据说明
- **依据要求**：不能是黑箱结论，必须说明参考了哪几条历史审批记录（如"参考了 2024-06-15 工单 #TKT-123 的同类型破损处理：审查通过，赔付 800 元"）
- **呈现方式**：审批人可见"AI 建议审批意见"，标注 **"AI 建议，需人工确认"**，审批人仍需手动操作通过/拒绝
- **约束**：AI 不能自动执行审批，仅做参考展示

### 2.3 扫描品控：AI 辅助识别异常子类型与严重度（加分项）

- **输入**：扫描描述文本 + 品控规则列表 +（如有图片）图片描述
- **输出**：推荐的品控异常子类型 + 严重度等级 + 置信度
- **标注要求**：明确标注 **"AI 建议，需人工确认"**

## 3. Prompt 设计思路

### 3.1 核心设计原则

- **角色设定**：将模型设定为"运单异常处理辅助助手"，限定在物流/仓储领域
- **结构化输出**：要求模型返回明确的 JSON 格式，包含 `suggestion`、`confidence`、`reasoning` 字段
- **约束声明**：Prompt 中明确强调"你的建议仅供参考，最终决定权在人工审批人"

### 3.2 异常分类 Prompt 示例结构

```
你是一个运单异常处理辅助助手。
根据以下异常描述，从候选类型中选择最匹配的异常类型，并评估严重度。

异常描述：{description}
候选类型：{candidateTypes}
运单信息：{waybillSummary}

请返回 JSON：
{
  "suggestedType": "异常类型",
  "severity": "low | medium | high",
  "confidence": 0.0-1.0,
  "reasoning": "判断依据"
}
```

### 3.3 审批建议 Prompt 示例结构

```
你是一个运单异常审批辅助助手。
根据当前工单和历史审批记录，给出建议审批意见。

当前工单：
- 异常类型：{type}
- 异常描述：{description}
- 涉及金额：{amount}
- 上报时间：{reportedAt}

历史同类审批记录（最近10条）：
{historicalRecords}

请返回 JSON：
{
  "suggestedAction": "approve | reject | escalate",
  "confidence": 0.0-1.0,
  "reasoning": "判断依据",
  "referencedRecords": ["TKT-XXX: 同类型破损，审查通过，赔付800元", ...]
}
```

## 4. 安全与可靠性保障

### 4.1 "AI 建议需人工确认"原则

- 所有 AI 输出在前端必须以醒目样式标注 **"AI 建议，需人工确认"**
- AI 建议不能自动执行任何业务操作（审批、分类、放行等）
- 用户必须进行明确的人工操作（点击确认/修改后提交），AI 仅作为辅助参考

### 4.2 异常处理（阻塞性保障）

- AI 服务超时或调用失败**不能阻塞主流程**
- 上报/审批必须在 AI 不可用时也能正常走完完整流程
- 降级策略：AI 调用失败时，前端隐藏 AI 建议区域，不弹出错误提示，上报/审批功能不受影响
- AI 调用超时设置为 5 秒（短于主流程接口），不参与重试逻辑
